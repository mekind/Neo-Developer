# Stage 2: Clarification & Persona Design

## 서비스명: MyClaw

## 1. 서비스 약속 (Product Promise)

> **MyClaw — 비개발자가 대화만으로 자신만의 AI 에이전트를 정의하고, Gather town 스타일 공간에서 도트 캐릭터로 만나 대화하는 플랫폼**

---

## 2. 사용자 페르소나 (사용자)

### Primary Persona: 수진 (29, 스타트업 마케터)

- **Tech savviness**: 중간. Notion, Slack 능숙. 코딩은 못함
- **Goals**: 반복 업무(경쟁사 모니터링, 뉴스 요약) 자동화하고 싶음
- **Frustrations**: "개발자한테 부탁하기엔 너무 작은 일, 직접 하기엔 귀찮은 일"
- **Context**: 아침 출근길에 폰으로, 점심시간에 노트북으로
- **Quote**: "챗GPT는 매번 같은 거 물어봐야 해서 귀찮아. 내가 원하는 대로 알아서 해주는 애가 있으면 좋겠어."

#### Problem Story
수진은 매일 아침 경쟁사 3곳의 블로그를 확인한다. RSS 피드? 설정하다 포기했다. 결국 북마크 폴더 열고 하나씩 클릭한다. 새 글 있으면 Slack에 공유하는데, 이 과정이 매일 15분. "이거 봇이 해주면 좋겠는데" 생각은 하지만 "봇 만드는 건 개발자 일"이라고 생각해서 시도조차 안 함.

---

### Secondary Persona: 동현 (35, 1인 쇼핑몰 대표)

- **Tech savviness**: 낮음. 스마트스토어는 익숙, 나머지는 어려움
- **Goals**: 고객 문의 응대 시간 줄이기
- **Frustrations**: "챗봇 서비스 가입했는데 설정이 너무 복잡해서 결국 안 씀"
- **Context**: 배송 준비하면서 틈틈이, 주로 모바일
- **Quote**: "그냥 '배송 언제 와요' 물으면 '3일 걸려요' 답하는 거 하나만 있으면 되는데..."

#### Problem Story
동현은 카카오 채널 챗봇을 설정하려다 "시나리오 빌더"에서 막혔다. 조건 분기, 변수, 연동... 30분 헤매다 창 닫았다. 지금은 자주 묻는 질문을 메모장에 적어두고 복붙한다. 하루 20분 정도 반복 응대에 쓴다.

---

### Edge-Case Persona: 은서 (42, 회사원)

- **Tech savviness**: 낮음. 회사 시스템 외에는 새 도구 배우기 싫어함
- **Goals**: "그냥 재밌어 보여서" 들어옴. 뭔가 만들어보고 싶긴 함
- **Frustrations**: 뭘 만들어야 할지 모름. 아이디어가 없음
- **Context**: 퇴근 후 심심할 때, 호기심으로 접근
- **Quote**: "AI가 뭐든 해준다던데... 근데 뭘 시켜야 하지?"

#### Problem Story
은서는 친구가 "AI로 뭐든 만들 수 있대"라고 해서 접속했다. 빈 화면에 "에이전트를 만들어보세요"라고 써 있다. 뭘 만들어야 할지 모르겠다. 예시도 없고, 튜토리얼도 길어 보인다. 5분 만에 나갔다.

## 3. 두 개의 Agent 흐름 설계

### 3.1 온보딩 Agent (사용자 정보 Clarify)

**목적**: 새 사용자가 누구인지 파악하고 Memory.md에 저장

**질문 흐름**:

```
[시작]
"안녕하세요! 저는 여기서 당신을 도와줄 친구예요. 
먼저 당신에 대해 조금 알려주세요."

[Step 1: 기본 정보]
"어떻게 불러드릴까요?" 
→ 닉네임/이름 저장

[Step 2: 목적 파악]
"여기 오신 이유가 있으세요? 
예를 들면... 반복 업무 자동화, 정보 수집, 그냥 재밌어 보여서?"
→ 자유 응답 → 카테고리 분류

[Step 3: 관심 분야 (선택)]
"혹시 관심 있는 분야가 있으면 알려주세요. 
없으면 '넘어가기'라고 하셔도 돼요."
→ 저장 또는 스킵

[Step 4: 기술 수준 (암묵적 판단)]
질문 방식과 응답으로 추론. 직접 묻지 않음.

[완료]
"좋아요! {닉네임}님, 반가워요. 
이제 저 공간에서 에이전트 친구들을 만들어볼까요?"
```

**출력**: `Memory.md` (사용자 프로필 섹션)

---

### 3.2 Persona Builder Agent (에이전트 정의 Clarify)

**목적**: 사용자와 대화하며 에이전트의 Persona 정의 → persona.md, SOUL.md 생성

**핵심 원칙**:
- Tool로 명확히 처리 가능 → 안 묻고 진행
- 불명확 → 상냥하게 질문
- 비개발자도 "이런 거 해줘"만 말하면 정의되게
- **자연어 풀어서 말해도 의도 파악**

---

### 3.2.1 자연어 입력 처리 전략

사용자는 format 맞춰서 말 안 함. 다양한 형태로 들어옴:

| 입력 유형 | 예시 | 처리 전략 |
|----------|------|----------|
| **명확한 요청** | "경쟁사 블로그 알림 봇 만들어줘" | 바로 역할 추출 → 확인 후 생성 |
| **불만/문제 토로** | "아 맨날 블로그 확인하는 거 존나 귀찮아" | 문제 공감 → 해결책 제안 → 봇으로 연결 |
| **모호한 요청** | "뭔가 알아서 해주는 거 있으면 좋겠어" | 탐색 질문 → 구체화 |
| **예시 언급** | "친구가 쓰는 거 봤는데 그런 거" | "어떤 거였어요?" → 역추적 |
| **복합 요청** | "블로그도 보고 뉴스도 보고 그래줬으면" | 분리 → 하나씩 확인 |

---

### 3.2.2 의도 추출 흐름

```
[입력 분석]
사용자: "아 나 맨날 아침에 경쟁사들 블로그 확인하는데 
        이거 너무 귀찮거든? 그래서 뭔가 자동으로 알려주면 좋겠어"

[내부 파싱]
├─ 감정: 귀찮음, 반복 피로
├─ 현재 행동: 매일 아침 경쟁사 블로그 확인
├─ 원하는 것: 자동 알림
├─ 추출된 역할: "정보 수집 + 알림"
├─ 관련 Tool: insane-search (웹 크롤링)
└─ 빠진 정보: 어떤 경쟁사? 언제 알림?

[응답]
Builder: "아 매일 블로그 돌아보는 거 진짜 귀찮죠 ㅠㅠ
         제가 대신 확인해서 알려드릴까요?
         어떤 회사 블로그예요?"
```

### 3.2.3 대화 예시 - 자연어 처리

**Case 1: 불만 토로형**
```
사용자: "요즘 뉴스가 너무 많아서 뭐가 중요한지 모르겠어"

Builder: "맞아요, 정보 과부하 힘들죠.
         제가 중요한 것만 골라서 알려드릴까요?
         어떤 분야 뉴스요? IT? 경제? 아니면 전체?"

사용자: "IT쪽?"

Builder: "좋아요! 매일 IT 뉴스 중 핵심만 요약해드릴게요.
         아침에 출근할 때? 저녁에 퇴근할 때?"

사용자: "음 아침에"

Builder: "완료! 'IT뉴스봇' 만들었어요.
         매일 아침 8시에 핵심 뉴스 3개 알려드릴게요.
         저기 안경 쓴 로봇이 보이죠? 그 친구예요!"
```

**Case 2: 모호형**
```
사용자: "뭔가 도움되는 거 있으면 좋겠는데"

Builder: "좋아요! 몇 가지 물어볼게요.
         요즘 반복적으로 하는 일 중에 귀찮은 거 있어요?"

사용자: "음... 회의 일정 확인하는 거?"

Builder: "아 회의 일정! 캘린더에서 확인하시는 거예요?
         아침마다 '오늘 회의 뭐 있지?' 이런 거요?"

사용자: "응 맞아"

Builder: "그럼 매일 아침에 오늘 일정 알려주는 봇 만들까요?
         '오늘 회의 3개 있어요. 첫 번째는 10시에...' 이런 식으로요"

사용자: "오 좋아"

Builder: "캘린더 연결해드릴게요. 구글 캘린더 쓰세요?"
```

**Case 3: 복합 요청형**
```
사용자: "블로그도 보고 유튜브 새 영상도 알려주고 그랬으면 좋겠어"

Builder: "두 가지네요! 하나씩 만들까요, 아니면 합쳐서 하나로?
         
         1️⃣ 따로: 블로그봇 + 유튜브봇
         2️⃣ 합쳐서: 콘텐츠봇 (둘 다 알려줌)
         
         어떤 게 좋아요?"

사용자: "합쳐서"

Builder: "좋아요! 어떤 블로그랑 어떤 유튜브 채널이요?"
```

### 3.2.4 Persona Builder 범위 제한

**Persona Builder는 에이전트 정의 전용. 잡담/상담 봇 아님.**

```
[범위 안]
- 에이전트 만들기
- 에이전트 수정하기
- 에이전트 삭제하기
- 어떤 에이전트 만들지 고민 상담

[범위 밖 → 거절]
- 일반 잡담
- 에이전트와 관계없는 질문
- 감정 상담
```

**거절 예시**:
```
사용자: "오늘 날씨 어때?"
Builder: "저는 에이전트 만드는 것만 도와드려요!
         날씨 알려주는 봇 만들어드릴까요? 😊"

사용자: "요즘 힘들어..."
Builder: "저는 에이전트 만드는 걸 도와드리는 친구예요.
         혹시 기분 전환용 대화 친구 봇 만들어볼까요?"

사용자: "1+1 뭐야"
Builder: "그건 제 전문이 아니에요!
         계산해주는 봇 만들어드릴까요?"
```

**핵심**: 범위 밖 요청 → 공감 X, 거절 + 에이전트 생성으로 유도

---

### 3.2.5 의도 추출 체크리스트

```
□ 에이전트 정의 관련 요청인가?
  ├─ Yes → 진행
  └─ No → 거절 + 에이전트 생성 유도

□ 역할이 명확한가?
  ├─ 명확 → Tool 매칭
  └─ 모호 → "~하는 거 맞아요?" 확인

□ 필수 정보가 있나?
  ├─ 대상 (뭘 볼지): 있음/없음
  ├─ 주기 (언제 알릴지): 있음/없음/기본값 사용
  └─ 채널 (어디로 알릴지): 있음/없음/기본값 사용
```

**질문 흐름**:

```
[시작]
"새 에이전트를 만들어볼까요? 
어떤 일을 해주는 친구를 원하세요?"

[Step 1: 핵심 역할]
사용자: "경쟁사 블로그 새 글 알려주는 봇"
→ 역할 추출: "정보 수집 + 알림"
→ 관련 Tool 자동 매칭: insane-search

[Step 2: 트리거/주기] (필요시만)
"언제 알려드릴까요? 
매일 아침? 새 글 올라오면 바로?"
→ cron 설정 또는 이벤트 트리거

[Step 3: 전달 방식] (필요시만)  
"어디로 알려드릴까요?
Discord? Slack? 아니면 여기서 직접?"
→ 채널 설정

[Step 4: 성격/말투] (선택)
"이 친구가 어떤 느낌이면 좋겠어요?
친근하게? 깔끔하게? 아니면 기본으로 할까요?"
→ 성격 파라미터

[Step 5: 외형] (선택)
"어떻게 생겼으면 좋겠어요?
간단히 설명해주시면 도트 캐릭터로 만들어볼게요.
예: 파란 모자 쓴 고양이"
→ persona.md의 이미지 프롬프트

[확인]
"정리해볼게요:
- 이름: {자동생성 또는 사용자 지정}
- 역할: {역할 요약}
- 주기: {주기}
- 성격: {성격}
- 외형: {외형 설명}

이대로 만들까요?"

[완료]
→ persona.md, SOUL.md, skill 연결 설정 생성
```

## 4. 출력물 스키마

### 4.1 Memory 구조 (llm-wiki 패턴)

```
~/.myclaw/users/{user_id}/
├── index.md          # 이 사용자의 모든 문서 목록
├── log.md            # append-only 변경 이력
├── profile.md        # 사용자 프로필
├── preferences.md    # 선호도 설정
├── agents/           # 이 사용자가 만든 에이전트들
│   ├── index.md      # 에이전트 목록
│   └── {agent_id}/
│       ├── persona.md
│       ├── SOUL.md
│       └── config.md
└── context/          # 대화에서 축적된 컨텍스트
    ├── interests.md  # 관심사
    └── history.md    # 주요 이벤트
```

#### index.md (사용자 문서 허브)

```markdown
# {닉네임}'s Space

## Profile
- [[profile]] - 기본 정보
- [[preferences]] - 선호도

## Agents
- [[agents/news-bot-001/persona|뉴스봇]] - 경쟁사 블로그 모니터링
- [[agents/reminder-001/persona|리마인더]] - 할 일 알림

## Context
- [[context/interests]] - 관심사: 마케팅, 트렌드
- [[context/history]] - 활동 이력
```

#### profile.md

```markdown
---
type: user-profile
created: 2026-04-26
updated: 2026-04-26
links:
  - [[preferences]]
  - [[context/interests]]
---

# Profile

## Identity
- name: 수진
- joined: 2026-04-26

## Context
- purpose: 반복 업무 자동화
- tech_level: medium (추론됨)

## Related
- 선호도: [[preferences]]
- 관심사: [[context/interests]]
- 만든 에이전트: [[agents/index]]
```

#### log.md (append-only)

```markdown
# Activity Log

<!-- 최신 항목이 위로 -->
- 2026-04-26 14:30: 에이전트 "뉴스봇" 생성 → [[agents/news-bot-001/persona]]
- 2026-04-26 14:00: 온보딩 완료 → [[profile]] 생성
- 2026-04-26 13:55: 첫 접속
```

### 4.2 Persona Builder의 Memory 활용

Persona Builder가 에이전트 정의 시 Memory를 읽어서 활용:

```
[Memory 참조 흐름]

1. 사용자가 "경쟁사 블로그 알려주는 봇 만들어줘" 요청

2. Persona Builder가 Memory 참조:
   ├─ [[profile]] 읽기 → tech_level: medium
   ├─ [[context/interests]] 읽기 → 마케팅, 트렌드
   └─ [[preferences]] 읽기 → tone: friendly, notification: morning

3. 자동 추론:
   - "경쟁사 블로그" + interests(마케팅) → 마케팅 블로그 특화
   - notification: morning → 기본 알림 시간 오전 8시
   - tone: friendly → 에이전트 말투도 친근하게

4. 질문 최소화:
   "마케팅 관련 블로그를 모니터링하는 거죠?
   매일 아침에 알려드릴까요?"
   → 이미 아는 건 확인만, 모르는 것만 질문
```

**핵심**: Memory에 있는 정보는 다시 안 물어봄. 문서 간 link로 연결된 컨텍스트를 읽고 활용.

## 6. User Journey & Scenarios

### 6.1 수진 (Primary) - User Journey

| Stage | 상황 | 행동 | 감정 | 마찰 | 기회 |
|-------|------|------|------|------|------|
| **Trigger** | 또 경쟁사 블로그 3개 돌며 15분 날림 | "이거 자동화 안 되나" 검색 | 짜증 + 기대 | 대부분 솔루션이 개발자용 | "코딩 없이 가능" 강조 |
| **First contact** | cmux 발견, 접속 | 메인 화면 둘러봄 | 호기심 + 경계 | "이것도 복잡하면 어쩌지" | 30초 안에 가치 보여주기 |
| **Onboarding** | 온보딩 Agent와 대화 | 닉네임, 목적 답변 | 편안함 | 질문 많으면 이탈 | 3-4개 질문으로 끝내기 |
| **Core usage** | Persona Builder로 봇 생성 | "경쟁사 블로그 알려줘" 입력 | 기대감 | 원하는 대로 안 되면 실망 | 첫 봇이 바로 동작해야 함 |
| **Repeat loop** | 매일 아침 알림 받음 | 알림 확인, 가끔 수정 요청 | 만족 | 알림이 안 오거나 쓸모없으면 불신 | 정확도 + 커스터마이징 |
| **Churn risk** | 한 달 후 | "이거 없어도 되나?" 생각 | 무관심 | 새 가치 없으면 이탈 | 새 봇 추천, 활용 팁 |

#### 수진 시나리오

**S1: 첫 봇 생성 (Happy Path)**
```
Given: 수진이 온보딩 완료 후 Gather town 공간에 입장
→ 느낌: 신기함, "여기서 뭐 하지?"
→ 행동: [+] 버튼 클릭 → Persona Builder 캐릭터에게 다가감
→ 대화: "경쟁사 블로그 새 글 알려주는 봇 만들어줘"
→ 시스템: "어떤 블로그요? URL 알려주시거나 회사 이름 말씀해주세요"
→ 대화: "A사, B사, C사 블로그"
→ 시스템: "찾았어요! 매일 아침 8시에 알려드릴까요?"
→ 대화: "응"
→ 시스템: "뉴스봇 만들었어요! 저기 파란 부엉이가 보이죠? 내일부터 알려드릴게요"
→ 느낌: 뿌듯함, "진짜 이렇게 쉽네"
```

**S2: 알림 수정 (Alternative Path)**
```
Given: 뉴스봇이 아침 8시에 알림을 보냈는데 출근 중이라 못 봄
→ 느낌: 아쉬움
→ 행동: 뉴스봇 캐릭터에게 다가감
→ 대화: "알림 시간 좀 바꿀 수 있어?"
→ 시스템: "물론이요! 몇 시가 좋으세요?"
→ 대화: "점심시간 12시"
→ 시스템: "바꿨어요! 내일부터 12시에 알려드릴게요"
→ 느낌: 편함, "말만 하면 되네"
```

**S3: 봇이 이상하게 동작 (Error Path)**
```
Given: 뉴스봇이 관련 없는 글도 알려줌
→ 느낌: 짜증
→ 행동: 뉴스봇에게 다가감
→ 대화: "아니 이건 경쟁사 글이 아닌데?"
→ 시스템: "앗 죄송해요! 어떤 글이 잘못 왔는지 알려주시면 더 정확하게 할게요"
→ 대화: "이 URL은 A사 아니야"
→ 시스템: "확인했어요. A사는 이 도메인만 볼게요. 다시 필터 설정했어요!"
→ 느낌: 일단 납득, "고쳐지면 OK"
```

### 6.2 동현 (Secondary) - User Journey

| Stage | 상황 | 행동 | 감정 | 마찰 | 기회 |
|-------|------|------|------|------|------|
| **Trigger** | 또 "배송 언제 와요" 복붙 | "이거 자동화하면 좋겠다" | 피로 | 챗봇 서비스 가입했다 포기한 경험 | "설정 없이 바로" 강조 |
| **First contact** | 광고나 지인 추천으로 접속 | "또 복잡하면 안 써" 경계 | 의심 | 첫 화면이 어려워 보이면 바로 이탈 | 극도로 단순한 진입 |
| **Onboarding** | 대화로 진행 | 최소한만 답변 | 귀찮음 | 질문 2개 넘으면 "됐어" | 2개 질문으로 끝 |
| **Core usage** | 봇 생성 | "배송 문의 답변 봇" 요청 | 반신반의 | "내 상황에 맞게 되나?" | 템플릿 + 커스텀 |
| **Repeat loop** | 고객이 봇과 대화 | 지켜보기 | 안도 | 봇이 이상한 답변하면 민망 | 사람이 개입할 수 있는 옵션 |
| **Churn risk** | 봇 응답이 부정확할 때 | "그냥 내가 할게" | 포기 | 고치는 것도 복잡하면 이탈 | 자연어로 수정 가능 |

#### 동현 시나리오

**S4: 단순 응대 봇 생성**
```
Given: 동현이 처음 접속, 온보딩 중
→ Onboarding: "어떻게 불러드릴까요?"
→ 대화: "동현"
→ Onboarding: "어떤 걸 해결하고 싶으세요?"
→ 대화: "배송 문의 답변"
→ Onboarding: "바로 만들어볼까요?"
→ 대화: "응"
→ 시스템: Persona Builder로 연결
→ Builder: "고객이 뭐라고 물으면 뭐라고 답하면 될까요?"
→ 대화: "'배송 언제 와요'면 '주문 후 2-3일 걸려요'라고 해줘"
→ Builder: "알겠어요! 다른 질문은요?"
→ 대화: "'교환 가능해요?'면 '7일 이내 가능해요'라고"
→ Builder: "두 가지 답변 봇 만들었어요. 더 추가하고 싶으면 언제든 말씀하세요!"
→ 느낌: "진짜? 이게 끝?"
```

**S5: 봇 응답 수정**
```
Given: 고객이 "반품 어떻게 해요?"라고 물었는데 봇이 "잘 모르겠어요"라고 답함
→ 느낌: 당황
→ 행동: 봇에게 다가감
→ 대화: "반품 질문하면 '카톡으로 연락주세요'라고 해줘"
→ 시스템: "추가했어요! 이제 반품 질문에 답할 수 있어요"
→ 느낌: "이렇게 쉽게 고쳐지네"
```

### 6.3 은서 (Edge-case) - User Journey

| Stage | 상황 | 행동 | 감정 | 마찰 | 기회 |
|-------|------|------|------|------|------|
| **Trigger** | "AI로 뭐든 된다던데" 호기심 | 검색 후 접속 | 기대 + 막막함 | 뭘 만들지 모름 | 예시로 시작 |
| **First contact** | 빈 화면 or 복잡한 옵션 | "뭘 해야 하지?" | 혼란 | 5분 안에 뭔가 안 되면 이탈 | 가이드 캐릭터가 먼저 말 걸기 |
| **Onboarding** | 대화 시작 | "뭘 만들 수 있어?" 질문 | 탐색 | 설명이 길면 이탈 | 예시 3개 → 선택 |
| **Core usage** | 예시 선택 or 간단한 것 시도 | "대화 친구" 선택 | 재미 | 기대만큼 안 되면 실망 | 첫 경험이 "와 신기하다" |
| **Repeat loop** | 가끔 들어와서 대화 | 심심할 때 접속 | 소소한 즐거움 | 새로움 없으면 안 옴 | 봇이 먼저 말 거는 기능 |
| **Churn risk** | "뭐 더 할 게 없네" | 접속 안 함 | 무관심 | 활용법 모름 | 새 활용법 추천 |
