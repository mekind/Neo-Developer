# Stage 5: Task Plan & Validation Design

## 1. 실행 계획 (모듈 기반)

### Module 1: Memory 시스템
**목적**: 사용자/에이전트 정보 저장 구조

| Task | 설명 | 산출물 |
|------|------|--------|
| M1-1 | 폴더 구조 정의 | `~/.cmux/users/{user_id}/` 구조 |
| M1-2 | profile.md 스키마 | 사용자 프로필 템플릿 |
| M1-3 | index.md 허브 | 문서 링크 관리 |
| M1-4 | agents/ 구조 | 에이전트별 폴더 구조 |

**검증**: 파일 생성/읽기/링크 동작 확인

---

### Module 2: 온보딩 Agent
**목적**: 신규 사용자 정보 수집 → Memory 저장

| Task | 설명 | 산출물 |
|------|------|--------|
| M2-1 | 온보딩 Agent 정의 | SOUL.md (온보딩용) |
| M2-2 | 질문 흐름 구현 | 닉네임 → 목적 → 관심분야 |
| M2-3 | Memory 저장 연동 | profile.md 생성 로직 |
| M2-4 | 완료 후 공간 진입 | Gather 공간으로 이동 |

**검증**: 온보딩 완료 → profile.md 생성 확인

---

### Module 3: Persona Builder Agent
**목적**: 자연어 → 에이전트 정의 생성

| Task | 설명 | 산출물 |
|------|------|--------|
| M3-1 | Persona Builder 정의 | SOUL.md (Builder용) |
| M3-2 | 의도 추출 로직 | 자연어 → 역할/대상/주기 파싱 |
| M3-3 | Tool 매칭 로직 | 역할 → skill 연결 |
| M3-4 | clarify 질문 생성 | 빠진 정보 질문 |
| M3-5 | persona.md 생성 | 외형 설명 템플릿 |
| M3-6 | SOUL.md 생성 | 성격/인사말/행동규칙 |
| M3-7 | config.md 생성 | skill/스케줄/채널 설정 |
| M3-8 | 범위 밖 거절 | 에이전트 생성 유도 |
| M3-9 | Memory 참조 | 기존 정보 활용 |

**검증**: 다양한 자연어 입력 → 올바른 에이전트 정의 생성

---

### Module 4: 에이전트 런타임
**목적**: 생성된 에이전트가 실제로 동작

| Task | 설명 | 산출물 |
|------|------|--------|
| M4-1 | SOUL.md 로드 | 성격/행동규칙 적용 |
| M4-2 | 인사말 랜덤 픽 | greetings_alert/approach |
| M4-3 | Tool(Skill) 실행 | insane-search 등 연동 |
| M4-4 | 범위 밖 거절 | "내 전문이 아니야" |
| M4-5 | 에이전트 최대 3개 제한 | 초과 시 안내 |

**검증**: 에이전트에게 질문 → 역할에 맞는 응답

---

### Module 5: Gather 공간 UI
**목적**: 캐릭터 이동 + 에이전트 상호작용

| Task | 설명 | 산출물 |
|------|------|--------|
| M5-1 | 공간 렌더링 | 기본 맵 |
| M5-2 | 사용자 캐릭터 이동 | WASD/화살표 |
| M5-3 | 에이전트 캐릭터 배치 | 도트 캐릭터 표시 |
| M5-4 | 근접 시 대화 시작 | 다가가면 채팅창 |
| M5-5 | [+] 버튼 | Persona Builder 호출 |

**검증**: 이동 → 다가감 → 대화 시작 → 응답 받음

## 2. 검증 케이스 (Validation Cases)

### VC-01: 온보딩 Happy Path
```
Given: 신규 사용자 첫 접속
When: 온보딩 Agent와 대화 (닉네임: "수진", 목적: "반복업무 자동화")
Then: 
  - profile.md 생성됨
  - profile.md에 name: 수진, purpose: 반복업무 자동화 포함
  - Gather 공간으로 이동됨
```

### VC-02: 에이전트 생성 - 명확한 요청
```
Given: 온보딩 완료된 사용자
When: Persona Builder에게 "경쟁사 블로그 새 글 알려주는 봇 만들어줘"
Then:
  - Builder가 "어떤 블로그요?" 질문
  - 사용자 답변 후 "언제 알려드릴까요?" 질문
  - 최종 확인 후 에이전트 생성
  - persona.md, SOUL.md, config.md 생성됨
  - Gather 공간에 캐릭터 등장
```

### VC-03: 에이전트 생성 - 자연어 풀어서
```
Given: 온보딩 완료된 사용자
When: "아 맨날 블로그 확인하는 거 너무 귀찮아 자동으로 알려주면 좋겠어"
Then:
  - Builder가 의도 파악 ("정보 수집 + 알림")
  - "대신 확인해드릴까요? 어떤 블로그요?" 질문
  - clarify 후 에이전트 생성
```

### VC-04: 에이전트 생성 - 모호한 요청
```
Given: 사용자가 뭘 만들지 모름
When: "뭐 만들 수 있어?"
Then:
  - Builder가 예시 제시 (알림봇, 대화친구, 리마인더 등)
  - 사용자 선택 후 해당 유형으로 clarify 진행
```

### VC-05: Persona Builder 범위 밖 거절
```
Given: Persona Builder와 대화 중
When: "오늘 날씨 어때?"
Then:
  - "저는 에이전트 만드는 것만 도와드려요!"
  - "날씨 알려주는 봇 만들어드릴까요?" (에이전트 생성 유도)
```

### VC-06: 에이전트 대화 - 역할 내
```
Given: 뉴스봇 에이전트 생성됨 (IT 뉴스 알림)
When: 사용자가 다가가서 "오늘 뉴스 뭐 있어?"
Then:
  - greetings_approach 중 하나로 인사 (랜덤)
  - IT 뉴스 검색 (insane-search)
  - 요약해서 전달
```

### VC-07: 에이전트 대화 - 역할 밖 거절
```
Given: 뉴스봇 에이전트
When: "1+1 뭐야?"
Then:
  - "그건 내 전문이 아니야 😅" (페르소나 말투에 맞게)
```

### VC-08: 에이전트 알림
```
Given: 뉴스봇 스케줄 설정됨 (매일 8시)
When: 8시 도달 + 새 뉴스 있음
Then:
  - greetings_alert 중 하나로 알림 ("오! 새 소식 있어!" 등)
  - 뉴스 요약 전달
```

### VC-09: Memory 참조
```
Given: 사용자 profile.md에 interests: 마케팅 저장됨
When: 새 에이전트 생성 시 "블로그 알림 봇 만들어줘"
Then:
  - Builder가 Memory 참조
  - "마케팅 관련 블로그요?" (이미 아는 정보로 추론)
  - 불필요한 질문 스킵
```

### VC-10: 에이전트 3개 제한
```
Given: 사용자가 이미 3개 에이전트 보유
When: 새 에이전트 생성 시도
Then:
  - "에이전트는 최대 3개까지 만들 수 있어요!"
  - "기존 에이전트를 삭제하거나, 나중에 더 만들 수 있는 요금제가 나올 예정이에요"
```

### VC-11: 인사말 랜덤 + 말투 일치
```
Given: 에이전트 SOUL.md에 formality: 반말, greetings_approach: ["왔어?", "오 안녕!", "뭐 찾아?"]
When: 사용자가 다가감
Then:
  - 3개 중 랜덤으로 하나 출력
  - 반말 유지
```

## 4. 실행 순서 (권장)

```
1. Module 1: Memory 시스템 (기반)
   └─ 폴더 구조 + 스키마 확정

2. Module 2: 온보딩 Agent
   └─ 첫 진입점, Memory 저장 연동

3. Module 3: Persona Builder Agent
   └─ 핵심 기능, 가장 복잡

4. Module 4: 에이전트 런타임
   └─ 생성된 에이전트 동작

5. Module 5: Gather 공간 UI
   └─ 시각적 통합
```
